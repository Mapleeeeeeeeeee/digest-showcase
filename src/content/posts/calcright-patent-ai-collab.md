用 AI 協作，把一個房東實務痛點做成 Android App，並取得新型專利
分享一個最近完成的里程碑。

我做的「算分明 CalcRight」Android App，目前已經開放測試；

背後的「租戶共用電錶的電費分攤系統」也正式取得中華民國新型專利。

新型第 M684527 號

租戶共用電錶的電費分攤系統

這個產品不是從技術題目開始，而是從一個很實務的問題開始。

我本身長期做租屋代管，分租套房、雅房、共生宅最常遇到的問題之一，就是共用電錶與水電費分攤。

同一個電錶，可能會遇到：

不同房間

不同入住日

不同退租日

不同帳單期間

不同房間狀態

不同費用分攤條件

以前很多房東靠 Excel、手算、截圖、LINE 訊息處理。

平常沒事的時候好像都可以。

但只要房客問一句：

「這個金額怎麼來的？」

就會發現真正困難的不是算出一個數字，而是這個數字能不能被說明、被保存、被追溯。

所以我想做的不是一個單純計算機，而是一套把房東每月流程串起來的工具：

案場設定

→ 抄表

→ 帳單輸入

→ 費用分算

→ 請款圖卡

→ PDF 備查

→ 歷史紀錄

這個過程中，AI 對我最大的幫助，不只是「幫我寫幾行程式碼」。

更重要的是，它協助我把一堆很混亂、很現場、很口語的實務經驗，整理成可以被開發、測試、修正、上架的產品文件。

例如：

1. 產品定位

2. 使用者痛點

3. 功能規格

4. 使用流程

5. UI 文案

6. 例外情境

7. 訂閱限制

8. 免費試用規則

9. 隱私權政策

10. 帳號刪除流程

11. FAQ

12. 官網產品資料

13. App Store / Google Play 審查回覆

14. 測試案例

15. 風險與免責口徑

這次我最大的體會是：

AI 開發真正有價值的地方，不是把「一句需求」直接變成「一個 App」。

而是把領域經驗一步一步轉成：

可以討論的需求

可以拆解的規格

可以執行的任務

可以驗收的測試

可以上線的產品

我自己不是把 AI 當成單純寫 code 的工具，而是把它當成產品協作夥伴。

整個流程大概是：

實務問題

→ GPT 協助整理產品邏輯

→ Codex / Claude Code 拆解與修改程式

→ 回到 App 測試

→ 發現問題

→ 再整理成規格

→ 再交給 AI 修正

→ 再驗收

其中最容易被低估的，其實是「規格文件」。

如果規格不清楚，AI 真的可以很快幫你做出東西，但那個東西很可能不是你真正要的。

尤其像訂閱制 App，很多問題不是畫面做出來就好。

例如：

免費試用到底怎麼算？

刪除案場後能不能重新試用？

換手機後資料怎麼回來？

取消訂閱後資料會不會不見？

帳號刪除時本機資料怎麼處理？

哪些資料能上雲？

哪些資料只能留在本機？

哪些文案不能過度承諾？

這些如果沒有先講清楚，程式做得出來，但產品口徑會亂。

以「算分明」來說，最後我們整理出一個比較清楚的產品原則：

算得清楚。

說得明白。

紀錄留得住。

也就是說，這個 App 不是只追求「算出答案」，而是要讓房東每個月的費用分算、請款與備查流程變得比較標準化。

對我來說，這次專利與 Android 測試版開放，不只是產品里程碑，也是一個 AI 協作開發的實驗結果。

我的結論是：

AI 可以大幅加速開發。

但 AI 不能取代你對真實問題的理解。

真正的產品問題，還是要從現場長出來。

AI 最強的地方，是把你腦中混亂但有價值的經驗，整理成可以被執行的系統。

目前 Android 版已經開放測試，接下來會繼續優化 UI、官網、測試流程與正式上架文案。

如果你也在用 Claude Code、Codex 或其他 AI 工具做產品，我會建議不要只問：

「幫我寫這個功能。」

而是先問：

「這個問題有沒有被定義清楚？」

「這個流程有沒有例外情境？」

「這個功能做出來後要怎麼驗收？」

「使用者看到的文案會不會誤解？」

「產品承諾會不會超過實際能力？」

寫 code 很快。

把問題講清楚，才是最難也最重要的部分。
