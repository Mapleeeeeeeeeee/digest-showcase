【開源】Get-a-Life 工具（曼陀羅）的開發分享

各位好，先前有和大家分享過一款曼陀羅工具的開發，這邊先提供成品的 github 連結，可以參考瀏覽：https://github.com/dowiosamuelu/get-a-life

除了成品外，我也分享幾個我在開發時的習慣與決策，希望對大家有幫助。但有個補充說明：

這款工具的「技術實踐方式」並不一定是最佳解，主要是因為 (1) 本來只是自用的小專案 (2) 我的手刻程式經驗僅限於靜態網頁與簡單的 python，對伺服器、網頁後端的較不熟悉。

所以很歡迎大家切換成自己習慣的技術，實作出不同的版本！

---

✴︎ 開發的想法（點子從哪裡來的？）

我的好朋友和我分享，他最近在用「曼陀羅思考法」去盤點生活。但我天生不太喜歡「照著別人的形式來」，於是當時就想：我想找到其他方法⋯⋯

不過深入分析自己的需求後，發現曼陀羅（或者說一張人生地圖）還是有幾個吸引我的地方：

* 當別人問起自己是誰時，可以有良好的定位。

* 當自我懷疑的時候，可以看清自己的特質與曾經的決策。

* 當有新的事情可以投入時，可以回頭來看是否符合自己的人生規劃。

所以原先的曼陀羅沒有不好，但可以更客製化。從上面的條列可以抓出幾個關鍵屬性：時間軸、盤點、行動規劃。這些也成為我設計工具時的重要依據。

在分析需求時，我也發現我會把人生定調成一場「冒險」，所以如果將這個工具加上「地圖」與「佔領」的元素，也會讓我更有動力使用。

→ 最後整體定調：我要把曼陀羅的格子當作「土地」，並透過思考或實際行動來維護它們。

✴︎ 成品的類型

原先想做 Obsidian 插件，可以將筆記節點化 link 到 64 個小格子上，更有探索宇宙的風格。但有幾個缺點：(1) 連結後不一定看得出時間順序 (2) 只是小進展也要寫成筆記 (3) 只能透過 link 的多寡去判斷格子的狀態，而非新鮮程度。

→ 最後參考了我自己的另一套日記系統，設計成網頁的形式，並讓我的介面與互動形式可以與「探險」、「耕耘」、「進度」相關。

✴︎ 開發過程——限制成品的形式，盡可能符合使用習慣

這次的開發過程，主要切成幾個階段，每段都不太長：

* 先建置出最基本的介面，也就是 9×9 方格，以及整體的佈景主題

* 確認我的「進度」文字要用怎樣的方式儲存

* 加上直接在介面中登記進度的功能

* 處理進度文字在介面中的顯示

* 去做更新時間的土地深淺變化

* 加上「下一步」的功能

我想分享的是，在開發的過程中，不要忘記問自己：「你需要的是什麼？」

例如我最需要「看得清楚，並讓我願意持續使用」，那麼，驗證「介面是否美觀」對我來說就非常重要，且需要優先考慮。

開發時也會冒出很多支線，例如資料庫、月份瀏覽方式、截圖分享、開啟不同資料夾⋯⋯。但我始終抓著一個重點，就是——先做最需要的功能。不僅能避免專案過度膨脹，讓其易於維護，也會逼迫自己對這個工具負責，好好使用它。

（舉一個砍功能的例子：如果要修改格子的「標題」，我限定直接去 md 檔裡面修改，而沒有在介面上製作改名按鈕。一方面我本來就會用 vscode 開專案資料夾，所以編輯檔案不是問題；另一方面我也不希望這些名字時常被更換。於是就乾脆不做在介面裡了。）

✴︎ 介面設計與想像

這次的佈景主題，是參考我自己寫的另外一套筆記系統複製過來的。如果要從零開始，除了直接讓 claude 開始生成，也可以嘗試用這些方法：

1. 找自己喜歡的網站，請 AI 分析風格，或者告訴你重要的 css 設定。

2. 善用 UI 風格的關鍵字。像有一陣子我很喜歡 Neobrutalism，就可以用它為依據去跟 AI 溝通，修改你的介面。

3. 可以直接給 AI 設計建議與字型（額外分享：截圖中的字型是 justfont 的蘭陽明體）。例如這次專案中整體的色系、背景的等高線地圖、外側九宮格中間的小細框，都是我直接指定加上的。

雖然雕這些會花時間，但以後我會更願意使用，所以值得投資。

✴︎ 小補充：file over app 的哲學

我自己很喜歡 Steph Ango 提到的 file over app 概念，也就是希望自己的資料，可以不用依賴於某個特定的服務才能開啟、讀取（https://stephango.com/file-over-app）。

所以我把資料直接存成一個 md 檔，不走後端資料庫。這樣就算哪天我不再用這個工具，也能輕鬆把過去的紀錄打包、分析。
